# ☁️ AWS Cloud Infrastructure & Auto Scaling 실습 기록

본 문서는 AWS의 핵심 인프라 서비스(EC2, VPC, EBS)의 기본 운용부터 데이터 복구, 그리고 고가용성을 위한 Auto Scaling 구축 과정을 정리한 학습 기록입니다.

---

## 1. 주요 개념 정리

### 🖥️ EC2 (Elastic Compute Cloud)
- **정의**: AWS 클라우드에서 제공하는 가상 컴퓨팅 리소스로, 필요에 따라 성능과 용량을 조절할 수 있는 서버입니다.
- **AMI (Amazon Machine Image)**: 인스턴스를 실행하기 위한 OS, 애플리케이션 정보가 담긴 '붕어빵 틀'입니다. 백업 및 리전 간 설정 복사에 사용됩니다.

### 💾 EBS (Elastic Block Store)
- **정의**: EC2 인스턴스에서 사용하는 고성능 가상 하드디스크입니다.
- **Snapshot**: 특정 시점의 EBS 볼륨 상태를 저장한 백업본입니다. 이를 통해 데이터를 보존하거나 다른 리전으로 복제할 수 있습니다.

### ⚖️ Auto Scaling & Load Balancer
- **Auto Scaling**: 트래픽 변화에 따라 EC2 인스턴스 개수를 자동으로 조절하여 비용 절감과 서비스 안정성을 확보합니다.
- **Load Balancer (ALB)**: 들어오는 트래픽을 여러 대의 인스턴스로 분산시켜 서버 부하를 방지합니다.

## 2. 실습 진행 과정

### Step 1: 시드니 리전 인프라 구축 및 웹 서버 배포
1. **VPC 및 네트워크**: 독립된 가상 네트워크를 구축하고 IGW(인터넷 게이트웨이)를 연결하여 통신 환경을 조성했습니다.
2. **EC2 생성 및 Nginx 설치**: Ubuntu OS 기반 인스턴스를 생성하고, `apt`를 통해 Nginx 웹 서버를 가동했습니다.
3. **보안 설정**: 보안 그룹(Security Group) 인바운드 규칙에서 SSH(22), HTTP(80) 포트를 개방했습니다.

### Step 2: 리전 간 데이터 복구 (시드니 ➡ 서울)
1. **이미지화**: 시드니 서버를 AMI 및 스냅샷으로 생성했습니다.
2. **리전 복사**: 생성된 스냅샷을 서울(`ap-northeast-2`) 리전으로 복사했습니다.
3. **볼륨 복원**: 서울 리전에서 복사된 스냅샷을 기반으로 **볼륨**을 생성하고, 새 EC2 인스턴스에 **Attach**한 뒤 마운트하여 데이터를 성공적으로 이전했습니다.

### Step 3: Auto Scaling Group (ASG) 설정
1. **시작 템플릿**: 복사해온 AMI를 기반으로 인스턴스 생성 규칙을 정의했습니다.
2. **용량 구성**: 
   - **원하는 용량(Desired)**: 3 (기본 3대 유지)
   - **최소 용량(Min)**: 1
   - **최대 용량(Max)**: 5

## 🛠️ 주요 트러블슈팅 (Troubleshooting)

### ❌ `Permission denied (publickey)` & `type -1` 에러
- **원인**: 키 파일(`private-acc.pem`) 데이터가 손상되어 유효한 형식이 아니었으며, 접속 계정명 설정 오류가 있었습니다.
- **해결**: 
  - `chmod 400`으로 권한을 재설정하고, 텍스트 편집기를 통해 키 형식을 수동 복구했습니다.
  - OS에 맞는 정확한 사용자 이름(`ubuntu`)을 사용하여 접속을 성공시켰습니다.

### ❌ `Could not resolve hostname`
- **원인**: SSH 접속 명령어에 IP 주소가 아닌 인스턴스 ID를 입력하여 발생했습니다.
- **해결**: AWS 콘솔에서 **퍼블릭 IPv4 주소**를 확인하여 직접 입력함으로써 해결했습니다.
